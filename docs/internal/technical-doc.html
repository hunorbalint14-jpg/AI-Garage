Overview

scope · whole app

AI Garage is an AI‑powered, white-label SaaS for UK garages. It tracks MOT / service / VED dates, drafts and sends automated customer reminders across email / SMS / WhatsApp, captures bookings and payments, and pushes accounting data into Xero.

What it does

  • Vehicle & customer CRM — registration-keyed records, DVSA-enriched.
  • Reminder engine — MOT, service and road-tax reminders drafted by Claude and sent on a per-location schedule.
  • Bookings & jobs — bay-aware scheduling with Stripe Checkout pay-links.
  • Invoicing — issued in-app, optional Stripe pay-link, mirrored to Xero.
  • Campaigns — AI-drafted broadcast email / SMS to the whole customer list.
  • Mid-job quotes (DVI) — mechanic records a short video of extra work found, customer approves or declines via a token-gated link.
  • Audit log — append-only trail of staff-sensitive actions (settings, GDPR, financial, integrations) visible to owners/admins.
  • Multi-location orgs — owners manage many garages under one organisation.

System architecture

request flow

Every request enters through Vercel's edge, hits the Next.js proxy (which resolves the tenant from the host and refreshes the Supabase session cookie), then runs as a server action or route handler against Supabase — with calls out to Claude, Stripe, Xero, DVSA, Resend and Twilio as needed.

Design principle Two trust boundaries: user-scoped (RLS-enforced, used by pages and actions) vs. service-role (RLS-bypassing, used only inside webhooks and cron after the request is independently authenticated). The service-role client must never be reachable from a user-authenticated path.
03

Multi-tenancy model

subdomain · slug · RLS

Every garage gets a subdomain on the root domain. The proxy parses the host, attaches the slug to the request, and every page resolves the tenant from that slug before talking to the database. Tenant isolation is enforced by Postgres RLS, not by application code.

Naming & routing

The root domain comes from NEXT_PUBLIC_ROOT_DOMAIN. In development it's localtest.me:3000 (which resolves any subdomain to 127.0.0.1 without DNS changes); in production it's the real apex.

  • apex — marketing site, sign-up
  • <slug>.apex — tenant portals (staff + customer)
  • custom.example.com — optional white-label custom domain stored on organizations.custom_domain

Hierarchy

An organisation owns one or more locations. Almost every other row carries location_id — the location is the tenant boundary for data, the organisation is the tenant boundary for branding, billing and Xero / Stripe Connect.

organization (slug, name, primary_color, stripe_account_id, xero_*)

Tenant-scoped data access

Two clients are used inside the same request:

  • User-scoped (createClient() from lib/supabase/server.ts) — used for all reads / writes after context is established. RLS does the isolation.
  • Service-role (createAdminClient()) — used only for the initial location lookup in staff-context.ts (the slug is not sensitive and membership is independently verified afterwards), and inside webhooks / cron.

RLS policies key off the is_garage_staff(location_id) helper function, which is security definer and stable, so the planner can hoist it. Customer-facing rows additionally allow auth.uid()-matched self reads.

04

Authentication & sessions

supabase auth · webauthn

Authentication uses Supabase Auth (email + password) with optional WebAuthn passkeys for re-authentication. Sessions are cookie-based and have a hard 12-hour absolute timeout on top of Supabase's own refresh window.

Identity model

  • auth.users — Supabase-managed. One per real human.
  • org_users — joins a user to an organisation with role owner or admin. Org-level access cascades to all locations under it.
  • location_users — joins a user to a single location with a role + permissions JSON. Used for per-location staff (e.g. a mechanic at one branch).
  • customers — non-staff users. Carry an optional user_id so a customer can sign in and use the customer portal.
  • webauthn_credentials — registered passkeys per user.

Session lifecycle

  1. Login server action calls Supabase Auth → cookies sb-* set.
  2. Proxy stamps an ai_session_started_at cookie on the first request after sign-in.
  3. On every subsequent request, proxy reads the stamp; if now − started > 12 h, calls signOut(), deletes sb-* cookies, redirects to /staff/login?expired=1 (or returns 401 for /api/*).
  4. Sign-out is a server action that calls signOut() and clears the stamp.

AI module · Claude

@anthropic-ai/sdk · haiku 4.5

Claude is used for all customer-facing copy. The integration is small and deliberately opinionated: one model (claude-haiku-4-5-20251001), tight system prompts, deterministic fallbacks if the API fails. There is no streaming; calls are synchronous inside server actions or cron handlers.

Surfaces

Use caseFile
Reminder email draftlib/ai-messages.ts · draftReminderMessage()
Reminder SMS draftlib/ai-messages.ts · draftSmsReminderMessage()
Custom 1-off messagelib/ai-messages.ts · draftCustomMessage()
Broadcast campaignlib/ai-messages.ts · draftBroadcastMessage()
Customer diagnostic chatlib/ai-diagnostic.ts
Voice → job sheetlib/ai-job-from-voice.ts
Labour estimationlib/ai-labour.ts

Prompt invariants

  • British English, <120 words for email, <130 chars for SMS.
  • Never include a phone number or URL — the booking link is appended programmatically.
  • Never ask the customer to call, reply, or get in touch off-channel.
  • Always start with "Hi FirstName," for reminders; "Dear customer," for broadcasts.
  • System prompts are cache_control: ephemeral for prompt-cache discount on the cron path.

If a call throws, the caller falls back to fallbackReminderMessage() — a deterministic template that hits the same compliance rules, so customers always get a coherent reminder.

Cost shape Most calls are ≤300 output tokens with a ~400-token cached system prompt. At Haiku rates this is a small fraction of a penny per reminder; for a 1000-customer garage doing a monthly MOT sweep, AI cost is dominated by Resend / Twilio per-message fees, not Claude.

Cron & reminder pipeline

vercel cron · hourly

A single Vercel cron pings /api/cron/tick every hour. That orchestrator finds enabled scheduled_tasks rows whose next_run_at is due, dispatches each to the right route handler with the task's location_id, then advances next_run_at using computeNextRunAt().

FIG 08·Areminder pipeline · per location
request async telemetry → DB per-location loop

Schedule storage

// scheduled_tasks row example
{
  task_type: "mot_reminders",
  enabled: true,
  frequency: "daily",
  hour: 9,
  day_of_week: null,
  next_run_at: "2026-05-20T09:00:00Z",
  settings: {
    remind_days_before: 30,
    channels: ["email", "sms", "whatsapp"]
  }
}

Dedupe & safety

  • 30-day dedupe per (vehicle, reminder_type, channel) — prevents accidental repeat sends.
  • Per-location channels — settings JSON controls which channels run for which reminder type.
  • Hard 100-row limit per location per tick — cron stays well under the 60s Vercel hobby budget.
  • Bearer auth — every cron handler requires Authorization: Bearer ${CRON_SECRET}; Vercel injects this from the project env when calling the configured cron path.
Failure mode If cron/tick partially fails (a single task's HTTP call errors), the orchestrator still advances next_run_at so we don't loop on the same failure. The error is recorded in the response payload and shows in Vercel cron logs. Channel-level failures (e.g. Twilio rejects a number) are written to reminders.status='failed' with the error message.

Payments & accounting

stripe connect · xero

Garages onboard their own Stripe Connect Express account through the platform. Every customer payment is a direct charge to that connected account with a platform fee skimmed (default 2%). Paid invoices and the corresponding payments are then synced to the garage's Xero tenant.

FIG 09·Ainvoice · pay · sync
HTTP browser redirect idempotent Xero write

Stripe Connect

  • Garage owner clicks Connect Stripe in /staff/settings → server action creates an Account Link, returns to /api/stripe/connect/return.
  • account.updated webhook keeps stripe_charges_enabled · stripe_payouts_enabled · stripe_details_submitted in sync on organizations.
  • Checkout sessions are created with the garage's stripe_account_id as the destination — funds settle to the garage; the platform fee is taken via application_fee_amount (STRIPE_PLATFORM_FEE_PERCENT, default 2%).
  • Bookings & invoices each have their own metadata, but the webhook handler treats them uniformly through session.metadata.invoice_id / booking_id.

Xero sync (lib/xero-sync.ts)

  • One Xero tenant per organization, attached via OAuth at /api/xero/connect/begin. Tokens (xero_access_token, xero_refresh_token, xero_token_expires_at) are stored on organizations.
  • ensureXeroContact() — idempotent contact upsert, stores xero_contact_id on customers.
  • pushInvoiceToXero() — converts job_items (or a single booking line) to Xero LineItem[], posts as ACCREC. UK VAT via OUTPUT2 + TaxExclusive.
  • pushPaymentToXero() — finds the first BANK account on the connected org, records a Payment against the invoice. Idempotent via xero_payment_id.

Webhook event matrix

Bank reconciliation (Xero Phase 2)

Once a garage's Stripe balance pays out to their real bank, payout.paid fires on the connected account. The handler posts a matching Receive Money bank transaction in their Xero so the accountant gets a ready-to-match line against the actual bank feed — reconciliation becomes one click. The payout id is the reference; the date is payout.arrival_date; the line is NoTax (VAT was already settled on the underlying invoices). Fees gap (gross customer payments minus net payout) accumulates as imbalance the accountant resolves as a periodic "Stripe fees" Spend Money. A dedicated Stripe clearing account is the natural Phase 3.

Idempotency contract Every Xero write is guarded by an "id already set" check before calling the API. Replays from Stripe (or from a manual webhook retry) are safe. The fast path for an already-synced payment is two DB reads and no Xero call.

Mid-job quotes (DVI)

video · token-gated · upsell

A mechanic working on a job who finds extra work records a short video, attaches line items + total, and sends the customer a token-gated link to approve or decline. Approved items get copied into job_items so the existing job → invoice → Stripe Connect path stays untouched.

Why it exists

  • Kills "I never agreed to that" disputes — the video + the approved quote are the audit trail.
  • Higher upsell conversion than a phone call — seeing the worn pad beats a verbal description.
  • Customer-friendly — token link works without an account; works on a bus stop on a phone.

Data model

  • job_quotes — lifecycle row (pending / approved / declined / expired / cancelled / approved_after_close), token hash, expiry, totals.
  • job_quote_items — frozen item snapshot — preserves what the customer agreed to even if products change later.
  • job_quotes_increment_view RPC — atomic view counter + first-view stamp.
  • Storage bucket job-quote-videos (private) — path format {location_id}/{job_id}/{quote_id}.{ext}, max 80 MB, MP4/MOV/WebM only.

Lifecycle

FIG 13·Aquote raise → customer decide → apply
HTTP / action privileged · service role email notification

Security posture

  • Token gating — 32 random bytes, base64url-encoded, only the sha256 hash is stored. Constant-time compare via crypto.timingSafeEqual. Same pattern as doc_shares.
  • Mint-once token — to resend, staff must cancel + recreate the quote. Matches the doc-shares posture and prevents accidental link leaks via "resend" buttons.
  • Atomic status transition — update … where status = 'pending' with a single predicate prevents two concurrent approvals from both inserting items.
  • Bucket privacy — no anon/authenticated policy on storage.objects for this bucket. All reads through 30-min service-role-minted signed URLs.
  • Job-state guard — if the job closed while the customer was deciding, the row flips to approved_after_close and staff are emailed to reopen or raise a separate job.
  • Cron expiry — /api/cron/quote-expiry sweeps stale pending rows hourly at :30.

Audit trail

Eight quote-specific audit actions are recorded: quote.create, quote.send, quote.cancel, quote.view (no actor — customer-side), quote.approve, quote.decline, quote.rebook (v2), quote.expire (cron-driven, no actor).

Payment posture Approval does not take payment. Items are added to the job and the existing job-complete → invoice → Stripe Connect pay-link path applies the final amount, because labour can overrun the quoted hours. Deposit-on-approve via Stripe Checkout is a deliberate v2 toggle.

Security & compliance

rls · gdpr · pci

Tenant isolation is enforced at the database. PCI is offloaded to Stripe Checkout (we never see card data). The platform takes a DPA acceptance per organisation and supports customer data deletion with an auditable log.

Application-level

  • RLS on every tenant-scoped table. Policy keys off is_garage_staff(location_id) (security definer) or auth.uid()-matched self-reads.
  • Service-role key only in lib/supabase/admin.ts; reachable only from webhooks, cron, and the tenant lookup in staff-context.ts.
  • 12-hour absolute session timeout in updateSession(), in addition to Supabase's refresh window.
  • Passkeys for high-privilege re-auth (settings changes, payments setup) — RP id is the root domain.
  • CRON_SECRET bearer on every cron handler.
  • Stripe + Resend signature verification on every webhook; signature failures return 400 with no DB side effect.
  • Stripe-hosted Checkout — card data never touches our infrastructure. PCI scope is SAQ A.
  • Doc share links — internal HTML docs gated by signed tokens at /docs/[slug]?t=<token>. Tokens are 32 random bytes; database stores only the SHA-256 hash. Constant-time compare via crypto.timingSafeEqual. Files live outside public/ so the gated route is the only way in. Owner-managed at /staff/docs; revoke / expire / view-cap built in.

Compliance hooks

  • DPA acceptance — organizations.dpa_accepted_at · dpa_version · dpa_accepted_by_user_id. Enforced via /staff/dpa-acceptance redirect.
  • Marketing consent — customers.marketing_email_consent · marketing_sms_consent · consent_updated_at. Broadcast send paths must check.
  • Right to erasure — anonymisation pass sets customers.anonymized_at; data_deletion_log records reason, requester and email hash.
  • Audit log — append-only audit_log table records ~35 actions (settings, GDPR ops, financial events, integrations, DVI quotes) with actor, IP, user-agent, and JSON metadata. RLS restricts reads to org owners/admins.
  • Data retention — organizations.data_retention_years drives scheduled cleanup of historical communications.
  • Cookie banner — components/cookie-banner.tsx; non-essential cookies gated.
  • Privacy policy — organizations.privacy_policy_url per tenant; /privacy hosts the platform-level policy.
